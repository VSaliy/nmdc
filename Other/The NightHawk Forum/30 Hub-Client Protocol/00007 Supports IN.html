Supports IN
« on: May 02, 2007, 10:11:56 PM »
This signals support for the $IN string
The $IN string is designed to replace the $MyINFO string.
Unlike the $MyINFO string, the $IN string has all parts seperated by a dollar sign. This allows for parts to be send stand-alone rather than the complete string as is the case with the $MyINFO string.
All parts on the $IN string have a 1-byte identifier, except for the first part which is the nickname.
The original tag as present in the $MyINFO string has its parts seperated by a comma (,), while in the $IN string, tag parts are seperated by a space.
Also, due to the integration of OP and Bot status in the status flag in the $IN string, $OpList and BotList/$BotList can be deprecated.
The status flag has been extended from a 4-bit value to a 32-bit value of which the first 8 have been defined.
The first 4 bits will be the same as the original status flag with:
bit 1   set to 1 by all clients by default
bit 2   set if user is Away
bit 3   set if user is server
bit 4   set if user is fireball
The next 5 bits are now defined and implemented in $IN as followed:
bit 5   set if user is OP
bit 6   set if $IN belongs to a Bot
bit 7   set if the client is in DND mode
bit 8   set if the client supports encryption
bit 9   set if the client supports partial search
Hereunder an example of a $MyINFO string and an $IN string during login:
$IN Daywalker$CDSL$F $S4799106616$TCZ++ V2.06 MA H2/4/20 S1 L30$|
$MyINFO $ALL Daywalker <Z++ V:2.06,M:A,H:2/4/20,S:1,L:30>$ $DSL $$4799106616$|
Next, an example of a client updating its hubcount and open slots for both $MyINFO and $IN strings
$IN [RO][RDS]robydjx$TH28/0/1 S29$|
$MyINFO $ALL [RO][RDS]robydjx <++ V:0.698,M:A,H:28/0/1,S:29>$ $0.005$$33594552321$|
